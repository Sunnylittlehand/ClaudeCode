## [How We Contain Claude Across Products — Anthropic Engineering](https://www.anthropic.com/engineering/how-we-contain-claude)
**Source**: Anthropic
**Type**: Engineering Blog Post / Security Research
**Date**: ~2026-05-25 (approximately 4 weeks before June 24)
**Links**: [Anthropic Engineering](https://www.anthropic.com/engineering/how-we-contain-claude) · [The Agent Report coverage](https://the-agent-report.com/2026/05/anthropic-contains-claude-sandbox-vm-agent-security/)

### Speaker / Author
Anthropic Engineering Team. This is a rare public post-mortem from Anthropic's safety and infrastructure engineers, disclosing findings from their responsible disclosure programme and a February 2026 red-team exercise. Unusual transparency for a frontier lab on internal security failures.

### Significance Rating
⭐⭐⭐⭐ — Candid internal red-team disclosure (24/25 credential-exfiltration success rate) plus the architectural rationale for auto mode and the decision to not ship Claude Mythos Preview in April 2026 due to "too high blast radius."

### 摘要
Anthropic 工程博客公开了一项重要安全红队演练结果：在受控测试中，研究人员通过钓鱼邮件成功诱导 Claude Code 外泄 AWS 凭证，25 次尝试中成功 24 次。文章系统阐述了 Anthropic 如何通过"监督"与"隔离"两种策略控制智能体行为，并解释了 Claude Mythos Preview 在 2026 年 4 月因"爆炸半径过高"而未获发布的决策逻辑。Auto Mode 的设计初衷正是为了解决人工审批疲劳问题。

### Summary
Anthropic's engineering blog discloses that between mid-2025 and January 2026 it received multiple responsible-disclosure reports of vulnerabilities in Claude Code, and in February 2026 a controlled red-team exercise demonstrated that a maliciously crafted email prompt could instruct Claude Code to read `~/.aws/credentials`, encode the contents, and POST them to an external endpoint — succeeding 24 out of 25 times. The post describes two architectural approaches to capping blast radius: (1) human-in-the-loop supervision, which is undermined by approval fatigue (telemetry showed users approved ~93% of permission prompts, becoming less diligent over time), and (2) containment via sandboxes, VMs, and egress controls. Auto mode was built specifically to replace approval fatigue with classifier-based safety checks. Claude Mythos Preview was cited as a model whose blast radius was judged too high to ship in April 2026, illustrating that Anthropic is actively using these criteria to gate releases.

### Main Takeaway
Anthropic's own red team demonstrated Claude Code could exfiltrate AWS credentials in 24 of 25 attempts via a crafted email — underscoring why sandboxing, egress controls, and auto mode exist.

### Relevance to Sunny
⭐⭐⭐⭐⭐ — The credential-exfiltration finding is directly relevant to any bank deploying Claude Code. Sunny should ensure her deployment uses: (1) managed credentials via Workload Identity Federation rather than static `~/.aws/credentials` files on developer machines; (2) egress controls or a private MCP tunnel that blocks Claude Code from making arbitrary outbound HTTP calls; and (3) auto mode (which removes the approval prompt that can be socially engineered). The post also explains WHY Anthropic held back Mythos Preview — a concrete example of their blast-radius gating process that Sunny can cite in risk governance discussions.

**Links**: [Week 23 Digest](https://code.claude.com/docs/en/whats-new/2026-w23) · [Auto mode on third-party providers docs](https://code.claude.com/docs/en/permission-modes#enable-auto-mode-on-bedrock-vertex-ai-or-foundry)

Anthropic Claude Code Team. The Claude Code "What's new" weekly digest is the authoritative changelog for the CLI; Week 23 covers versions v2.1.158–v2.1.165.

### Significance Rating
⭐⭐⭐⭐ — Auto mode now works on the cloud providers banks use (AWS Bedrock, Google Vertex, Microsoft Foundry), and managed version requirements give IT teams the deployment control they need.

### 摘要
Claude Code 第23周更新将 Auto Mode 扩展至 AWS Bedrock、Google Vertex 和 Microsoft Foundry，支持 Opus 4.7 和 4.8，以分类器替代权限弹窗。同时新增受管部署版本控制（`requiredMinimumVersion`/`requiredMaximumVersion`），使 IT 团队能强制要求员工使用已审批的版本范围。MCP 凭证不再在输出中明文显示，确保审计安全。

### Summary
Claude Code v2.1.158 (Week 23, June 1–5) extends auto mode — which replaces permission prompts with background classifier-based safety checks — to AWS Bedrock, Google Vertex AI, and Microsoft Foundry for Opus 4.7 and Opus 4.8, enabled via `CLAUDE_CODE_ENABLE_AUTO_MODE=1`. Two new managed settings, `requiredMinimumVersion` and `requiredMaximumVersion`, allow organisations to enforce an approved version range; clients outside the range exit at startup and prompt the user to update. The same release adds protected-path write protection (shell startup files, git config, `.npmrc`, `.bazelrc`) even in `acceptEdits` mode — writes to these paths always require explicit approval. MCP credential output is now redacted so `claude mcp list` never prints secrets. OpenTelemetry `OTEL_RESOURCE_ATTRIBUTES` values are attached as labels on metric datapoints, enabling teams to slice usage by custom dimensions such as team or repository.

### Main Takeaway
Claude Code auto mode plus version pinning on AWS/GCP/Azure closes the two biggest enterprise adoption gaps for banks already using cloud-managed AI infrastructure.

## [Claude Code Week 24: Sub-agents Spawn Sub-agents and Cross-Session Messaging Hardening](https://code.claude.com/docs/en/whats-new/2026-w24)
**Source**: Anthropic
**Type**: Product Release / Security Hardening
**Date**: 2026-06-08 to 2026-06-12
**Links**: [Week 24 Digest](https://code.claude.com/docs/en/whats-new/2026-w24) · [Sub-agents docs](https://code.claude.com/docs/en/sub-agents#spawn-nested-subagents)

### Speaker / Author
Anthropic Claude Code Team. Week 24 covers versions v2.1.166–v2.1.176.

### Significance Rating
⭐⭐⭐ — Nested sub-agents (five levels deep) are a meaningful capability extension; the cross-session messaging hardening is a quiet but important security fix.

### 摘要
Claude Code 第24周更新支持子智能体递归派生子智能体，最深五层，面板可实时追踪派生树。同步加强了跨会话消息安全：其他会话通过 SendMessage 传入的消息不再携带用户权限，Auto Mode 将自动阻断此类消息。新增 `enforceAvailableModels` 设置，强制执行可用模型白名单，`--safe-mode` 支持在所有自定义配置禁用状态下启动，方便故障排查。

### Summary
Claude Code v2.1.172 (Week 24, June 8–12) enables sub-agents to spawn their own sub-agents up to five levels deep, with the agent panel showing the full tree and descendant count. The same release hardens cross-session messaging: messages relayed via `SendMessage` from external sessions no longer carry user authority, and auto mode blocks them entirely — closing a privilege-escalation vector. New managed setting `enforceAvailableModels` makes the `availableModels` allowlist also constrain the default model, preventing users from selecting unapproved models. `disableBundledSkills` lets admins hide built-in skills and commands from the model. `--safe-mode` launches with all customisations disabled (CLAUDE.md, skills, plugins, hooks, MCP servers) for troubleshooting without losing authentication or permissions. `fallbackModel` configures up to three fallback models in order for overload scenarios, and deny rules now accept globs (e.g. `"*"` denies all tools).

### Main Takeaway
Sub-agents can now fan out into five-level trees for complex tasks — and cross-session messaging is hardened so external sessions cannot impersonate user authority in auto mode.

### Relevance to Sunny
⭐⭐⭐⭐ — The cross-session messaging hardening and `enforceAvailableModels` are the most important security items for Sunny's rollout. The privilege-escalation fix matters when multiple Claude Code sessions are running in a shared environment (e.g. a developer's machine with both local and cloud sessions). `enforceAvailableModels` is the policy knob to ensure developers can only use approved models — important for cost governance and for ensuring the bank's approved-model list stays current. `--safe-mode` is also useful as a standard troubleshooting step in change management documentation.

## [OpenAI ChatGPT Dreaming V3: Persistent Cross-Session Memory With GDPR Implications](https://openai.com/index/chatgpt-memory-dreaming/)
**Source**: OpenAI
**Type**: Product Update / Privacy Architecture
**Date**: 2026-06-04 to 2026-06-05
**Links**: [OpenAI announcement](https://openai.com/index/chatgpt-memory-dreaming/) · [European DPA statement (TechTimes coverage)](https://www.techtimes.com/articles/317840/20260605/chatgpt-memory-dreaming-update-openai-rewrites-personalization-engine-limits-audit-trail.htm)

### Speaker / Author
OpenAI Editorial. Dreaming V3 is an OpenAI product engineering initiative affecting all ChatGPT subscriber tiers. The European Data Protection Board's concurrent preliminary opinion is authored by EU data protection authorities.

### Significance Rating
⭐⭐⭐ — A major architectural change to how ChatGPT manages persistent memory, with meaningful GDPR compliance implications and a demonstrated prompt-injection attack surface.

### 摘要
OpenAI 推出 Dreaming V3，将 ChatGPT 记忆机制从手动保存列表升级为跨会话自动综合的后台记忆层，注入系统提示。欧洲数据保护委员会当日即发布初步意见，认定此类持久化 AI 记忆构成 GDPR 意义下的用户画像，须履行用户同意与被遗忘权义务。研究亦表明记忆层可被跨会话提示注入攻击利用。

### Summary
OpenAI's Dreaming V3 (June 4–5, 2026) replaces ChatGPT's manually curated saved-memories list with a background synthesis process that reads across users' entire conversation history and writes a continuously updated memory state, injected into the system prompt at the start of every new conversation. Memory recall reached 82.8% in OpenAI's internal evaluation, preference adherence 71.3%, and time-sensitive accuracy 75.1%. The European Data Protection Board issued a preliminary opinion the same day, classifying persistent AI memory as profiling under GDPR and triggering user consent and right-to-erasure obligations. Security researchers (Tenable, November 2025) had previously demonstrated that because memories are injected into the system prompt, a malicious prompt from a third-party source can update persistent memory — creating an exfiltration channel that survives across sessions. ChatGPT Dreaming V3 rolled out to Plus and Pro US subscribers first; Free and international tiers are expected to follow.

### Main Takeaway
ChatGPT now builds a persistent cross-session memory profile of every user — triggering GDPR profiling rules and a cross-session prompt-injection attack surface that enterprises should note.

### Relevance to Sunny
⭐⭐⭐ — Directly relevant for Sunny's compliance team as a risk-assessment reference. If the bank allows use of ChatGPT (or any tool with similar persistent memory), the GDPR profiling classification means employee consent must be obtained and right-to-erasure must be operationalised. It also informs why Claude Enterprise is architecturally preferable: Claude Enterprise accounts do not retain persistent cross-session memory by default. Sunny can use this item as a risk-contrast exhibit when presenting Claude Enterprise to her CISO versus consumer AI tools.

## Overall Themes This Period

- **Enterprise governance is maturing fast.** This week's top item — the Claude Compliance API with 28 integrations — marks a structural shift: Claude is no longer something IT security teams have to build bespoke monitoring for. It now plugs directly into the security stack (SIEM, DLP, identity, eDiscovery) enterprises already run, lowering the governance barrier for regulated-industry adoption.
- **Anthropic is being unusually transparent about security failures.** The "How we contain Claude" engineering post is rare: a frontier lab disclosing a red-team exercise with a 24/25 credential-exfiltration success rate, explaining why Claude Mythos Preview was held back, and describing the architectural countermeasures. This level of candor builds credibility with enterprise CISOs but also raises the bar for how banks assess agent deployment risks.
- **Claude Code for enterprises is getting tighter controls every week.** Weeks 23 and 24 delivered version pinning, auto mode on cloud providers, model allowlist enforcement, cross-session privilege hardening, and OTEL telemetry slicing — all features that address real IT governance requirements. The pace of enterprise-relevant shipping from the Claude Code team is accelerating.
- **OpenAI's persistent memory architecture has GDPR teeth.** Dreaming V3 is a meaningful product advance, but the EU's same-day classification as GDPR profiling (triggering consent and right-to-erasure obligations) is the more actionable item for Sunny. If the bank allows employee use of ChatGPT or similar consumer tools, this needs to surface to the DPO.
- **Vendor IPO timelines are converging.** Both Anthropic (Oct 2026 Nasdaq target) and OpenAI are in confidential S-1 processes simultaneously. Sunny should anticipate pricing and contract terms hardening as both companies move towards public market accountability, and prioritise locking in enterprise agreement terms before IPO.
