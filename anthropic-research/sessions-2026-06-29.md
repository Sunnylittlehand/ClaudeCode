## [Claude Code Trusted Devices: Admin-Required Device Verification for Remote Control Sessions](https://code.claude.com/docs/en/remote-control)

**Date**: 2026-06-26
**Links**: [Remote Control docs](https://code.claude.com/docs/en/remote-control) · [Claude Code changelog](https://code.claude.com/docs/en/changelog) · [Security analysis](https://www.penligent.ai/hackinglabs/claude-code-remote-control-security-risks-when-your-local-session-becomes-a-remote-interface/)

### Speaker / Author
Anthropic Engineering. This is a security hardening feature for Claude Code's Remote Control capability, which allows developers to monitor and steer local Claude Code sessions from a phone or browser.

### Significance Rating
⭐⭐⭐☆☆ — Meaningful enterprise security control that ties remote Claude Code access to enrolled, authenticated devices — directly addresses the risk that a stolen session cookie or compromised account could silently observe a running coding agent.

### 摘要
Anthropic为Claude Code远程控制功能推出"可信设备"（Trusted Devices）企业安全机制，现处于公测阶段。团队版和企业版管理员可强制要求成员在查看或操控远程Claude Code会话前先完成设备注册，注册时机与完整身份验证挂钩，有效防止账号凭证被盗后的会话劫持风险。

### Summary
Anthropic shipped Trusted Devices (beta) for Claude Code Remote Control on Team and Enterprise plans. When enabled by an admin, any member who wants to view or steer a local Claude Code session from claude.ai, the mobile app, or Claude Desktop must first enrol their device — enrollment is tied to a full authentication event, not a silent background process. The setting is off by default; owners must enable it in Claude Code admin settings. This closes a meaningful gap: previously, any signed-in account could access a remote session without proof of device possession.

### Main Takeaway
Claude Code's Remote Control feature now supports device-level access controls, closing the gap between "signed in" and "authorised device" for enterprise teams.

### Relevance to Sunny
⭐⭐⭐⭐☆ — Highly relevant for Sunny's FCA/PRA governance work. ## [Paving the Way for AI Agents in Biology: Anthropic VirBench Research](https://www.anthropic.com/research/agents-in-biology)
**Source**: Anthropic Research
**Type**: Research Paper / Essay
**Date**: 2026-06-08
**Links**: [Anthropic Research](https://www.anthropic.com/research/agents-in-biology) · [TechTimes coverage](https://www.techtimes.com/articles/318062/20260609/ai-agents-biology-are-too-inaccurate-use-anthropics-deterministic-tool-fix.htm)

### Speaker / Author
Laura Luebbert (Broad Institute / FutureHouse), published on Anthropic Research. Luebbert is a computational biologist working at the intersection of genomics and AI; she argues for deterministic tool design as a prerequisite for reliable scientific agents.

### Significance Rating
⭐⭐⭐☆☆ — Intellectually important research that applies beyond biology: shows current AI agents fail at seemingly simple structured-data retrieval tasks, and proposes a tool-design fix that lifts accuracy from 17% to 99.7%.

### 摘要
Anthropic研究显示，当前最优AI研究智能体在从NCBI病毒数据库检索病毒序列时准确率仅17-91%，且同一模型对相同查询给出不同答案。Anthropic与NCBI合作开发的确定性工具gget virus将准确率提升至99.7%，核心结论是：AI科学智能体的瓶颈不在模型能力，而在数据基础设施设计。

### Summary
Anthropic's research (published June 8) found that frontier AI agents — including Claude Sonnet 4 and GPT-5.5 — scored as low as 16.9% accuracy on a basic biology task (retrieving viral sequences from NCBI Virus) and gave inconsistent answers to the same query. The fix: Anthropic built `gget virus`, a deterministic tool that translates NCBI's web interface into a reliable programmatic call, lifting accuracy to 99.7%. The broader argument is that biological (and other structured) data infrastructure must be redesigned for agents rather than human browsers.

### Main Takeaway
AI agents aren't failing because models are weak — they're failing because the data plumbing was built for humans; building deterministic tool wrappers around existing APIs is the practical fix.

### Relevance to Sunny
⭐⭐⭐☆☆ — Useful background for Sunny's developer conversations. The paper's core lesson — that agent reliability depends heavily on the quality of the tools and APIs they call, not just the model — is directly applicable to Claude Code deployments in banking. When developers ask "why does Claude Code give inconsistent results?", this research provides a principled answer: the bottleneck is often tool/data infrastructure, not the model. It also supports the case for investing in well-designed MCP servers and structured data interfaces for internal bank systems.
